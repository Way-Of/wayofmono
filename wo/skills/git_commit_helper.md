# Git Commit Helper

This skill assists in creating high-quality, professional git commits that follow project standards.

## Workflow

### 1. Analysis
Observe the state of the repository to understand what has changed:
- \`git status\` to see modified files.
- \`git diff\` to review the actual code changes.
- Review the current \`implementation.md\` plan to understand the intent.

### 2. Grouping
Identify logical groups of changes. Commits should be atomic:
- Fixes and their corresponding tests should be together.
- Refactors should ideally be separate from new features.
- Documentation updates can be grouped with the features they describe or separate if they are significant.

### 3. Drafting
Draft a commit message using the imperative mood:
- **Good:** "Add user authentication logic"
- **Bad:** "Added user authentication logic"

Format:
\`\`\`
[Type]: [Short Description]

[Optional detailed body explaining 'why' not 'what']
\`\`\`
Types: \`feat\`, \`fix\`, \`docs\`, \`style\`, \`refactor\`, \`test\`, \`chore\`.

### 4. Execution
1. **Stage:** \`git add <specific_files>\` (avoid \`git add .\`).
2. **Commit:** \`git commit -m "<message>"\`.
3. **Verify:** \`git log -n 1\` to confirm.

## Rules
- **No AI Attribution:** Do not add "Generated by AI" or similar tags. The commit should look like it was written by the user.
- **Imperative Mood:** Always use the present tense ("Add", not "Added").
- **Atomic Commits:** Prefer multiple small, logical commits over one giant "catch-all" commit.
